# Harness-Agnostic Skill Design

This spec defines how skills in this repo are structured to work across multiple AI coding harnesses without duplication of core logic.

## Supported Harnesses

| Harness | Repo | SKILL.md format |
|---------|------|-----------------|
| Claude Code | https://github.com/anthropics/claude-code | Blockquote frontmatter (`> Key: value`) |
| opencode | https://github.com/anomalyco/opencode | YAML frontmatter (`--- name: / description: ---`) |

## Architecture

Each skill has **two thin entry-point files** and **shared phase/content files**:

```
skills/<skill-name>/
  SKILL.claude.md        # Claude Code entry point (blockquote frontmatter)
  SKILL.opencode.md      # opencode entry point (YAML frontmatter)
  SKILL.md               # gitignored — generated at install time
  install.sh             # copies the correct variant to SKILL.md
  .gitignore             # ignores SKILL.md
  phases/                # shared, harness-neutral workflow content
  agents/                # shared, harness-neutral agent definitions
  ...                    # formulas, scripts, etc.
```

`SKILL.md` is never committed. It is generated by `install.sh` (per-skill) or the repo-level `install.sh` (all skills).

## Entry-Point Files (SKILL.claude.md / SKILL.opencode.md)

Each entry-point file contains ONLY harness-specific content:

1. **Frontmatter** — in the format the harness expects
2. **Skill directory resolution** — search paths differ per harness
3. **Tool Mapping table** — maps abstract actions to harness-specific tool names
4. **Prerequisites** — where harness-specific checks differ (e.g., which project instructions files to look for)
5. **Phase references** — pointers to the shared phase files

Everything else (workflow logic, plan structure, templates, bash commands, agent prompts) goes in shared files.

## Tool Mapping

The entry-point file defines a Tool Mapping table. Shared phase files reference actions generically (e.g., "ask the user directly using the question/ask tool from the Tool Mapping") rather than naming harness-specific tools.

### Current mappings

| Action | Claude Code | opencode |
|--------|-------------|----------|
| Spawn sub-agent (full access) | `Agent` with `subagent_type="general-purpose"` | `task` tool with agent type `general` |
| Isolated experiment | `Agent` with `isolation="worktree"`, `mode="bypassPermissions"` | `bash`: `git worktree add` / `git worktree remove` |
| Ask user a question | `AskUserQuestion` | `question` tool |
| Enter/exit worktree | `EnterWorktree` / `ExitWorktree` | Manual via `bash` |
| Web search | `WebSearch` | `websearch` (requires Exa config) |
| Fetch URL | `WebFetch` | `webfetch` |
| Task tracking (built-in) | `TodoWrite` (NEVER use — use `bd`) | `todowrite` (NEVER use — use `bd`) |

## Shared Phase Files

Phase files in `phases/` describe WHAT to do, not WHICH tool to call. Rules:

- **Never reference a harness-specific tool name.** Say "spawn a sub-agent (per Tool Mapping)" not "use the Agent tool."
- **Bash commands are neutral.** Both harnesses execute bash identically — `bd`, `gh`, `git`, `uv run` are all fine.
- **Plan structure, templates, and bead commands are neutral.** These are the same regardless of harness.
- **Session boundary language is neutral.** Say "new session" not "new Claude Code session."

## Agent Files

Agent `.md` files in `agents/` should also be harness-neutral:

- Describe the agent's role, inputs, execution steps, and rules
- Reference `${SKILL_DIR}/` for paths to other skill files
- Avoid naming harness-specific tools — describe the action instead
- Use generic phrasing for built-in tool avoidance: "never use built-in todo/task tools" covers both `TodoWrite` and `todowrite`

## Project Instructions File

Both harnesses read `CLAUDE.md` natively. opencode additionally reads `AGENTS.md`, `opencode.md`, and `OPENCODE.md`.

- Claude Code prerequisites can require `CLAUDE.md` specifically
- opencode prerequisites should check all supported instruction file names
- Upstream config is persisted to `CLAUDE.md` since both harnesses read it

## SKILL_DIR Search Paths

| Harness | Paths searched |
|---------|---------------|
| Claude Code | `~/.claude/skills`, `/workspace/.claude/skills`, `.claude/skills` |
| opencode | `~/.claude/skills`, `~/.agents/skills`, `.claude/skills`, `.agents/skills`, `.opencode/skills`, `.opencode/skill` |

## Install Process

### Per-skill

```bash
cd skills/<skill-name>
./install.sh claude    # or: ./install.sh opencode
```

### Repo-wide

```bash
./install.sh claude    # installs all skills for Claude Code
./install.sh opencode  # installs all skills for opencode
```

The repo-level script generates `SKILL.md` for every skill, then copies them to the appropriate harness skills directory.

## Adding a New Harness

1. Add a column to the Tool Mapping table above
2. Create `SKILL.<harness>.md` for each skill with the new harness's frontmatter and tool names
3. Update `install.sh` scripts to accept the new harness name
4. Update SKILL_DIR search paths
5. Update prerequisite checks if the harness has different project instructions conventions

## Checklist for New Skills

- [ ] `SKILL.claude.md` with blockquote frontmatter and Claude Code tool mapping
- [ ] `SKILL.opencode.md` with YAML frontmatter and opencode tool mapping
- [ ] `.gitignore` containing `SKILL.md`
- [ ] `install.sh` that copies the correct variant
- [ ] Phase files in `phases/` with no harness-specific tool references
- [ ] Agent files in `agents/` with no harness-specific tool references
- [ ] Verify with: `grep -rn 'AskUserQuestion\|TodoWrite\|EnterWorktree\|ExitWorktree\|EnterPlanMode\|ExitPlanMode' phases/ agents/` — should return nothing
